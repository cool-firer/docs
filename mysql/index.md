# 聚簇索引 vs 二级索引（底层怎么存数据）

根据底层叶子节点存的内容不同，B+ 树在 MySQL 里分成了两种：

#### 1. 聚簇索引（Clustered Index / 主键索引）

- **特点**：索引和数据是在一起的。

- **叶子节点存什么**：存的是**完整的整行数据**。

- **数量**：一张表有且仅有一个聚簇索引（默认是主键，没主键就选第一个唯一索引，再没有 InnoDB 会自动生成一个隐式的 `row_id`）。

#### 2. 二级索引（Secondary Index / 辅助索引/普通索引）

- **特点**：比如你给 `age` 或者 `name` 字段建的索引。

- **叶子节点存什么**：不存整行数据，**只存“该索引字段的值” + “对应的主键 ID”**。

# 为什么是 B+ 树？（对比其他数据结构）

面试官很喜欢问：为什么不用二叉树、红黑树或者 B 树？

- **为什么不用红黑树/二叉树？**
  
  数据量大时，二叉树会变得“极高”，查一个数据可能要向下找几十层，每一次向下找都对应一次磁盘 I/O，太慢了。

- **为什么不用 B 树（B-Tree）？**
  
  B 树的**非叶子节点也会存整行数据**。这样导致一个 16KB 的节点里面塞不下几个索引，树会变高；而且 B 树的叶子节点没有双向链表，做范围查询时极其痛苦，需要不断地在树的上下层之间做中序遍历。

<br />

# 惊人的计算：3层 B+ 树能存多少数据？

在 MySQL 中，一个节点（一页）默认是 **16KB**。

- 假设主键是 `BIGINT`（8字节），指针是 6字节，那么非叶子节点的一条索引项大概是 14字节。一页可以存 $16\text{KB} / 14\text{B} \approx 1170$ 个指针。

- 假设底层叶子节点一条整行数据是 1KB，那一页可以存 16条数据。

- **高度为 3 的 B+ 树**可以存的行数约为：$1170 \times 1170 \times 16 \approx 2190万$ 行数据。

这意味着，在两千万级别的数据量下，你精准通过主键查询一条数据，**最多只需要 3 次磁盘 I/O**（甚至根节点常驻内存，只需要 1~2 次），这就是 B+ 树无可比拟的优势。

# 页的概念

“页”（Page）是 MySQL 最基础的**数据管理和传输单位**，它是数据库在“磁盘”和“内存”之间倒腾数据时的**集装箱**。

为了让你彻底搞明白，我们把它拆成两个维度来看：

### 1. 在磁盘上：它是文件的物理划分

当你在磁盘上建表、存数据时，MySQL 并不是按“行”为单位散落在硬盘上的，也不是把整个文件看作一个整体。它把磁盘上的数据文件（比如 `.ibd` 文件）整整齐齐地切成了一块一块，每一块的大小默认是 **16KB**。

- 磁盘上的这一个 16KB 的数据块，就叫一个**磁盘页（Disk Page）**。

- 你的 B+ 树索引结构、你存的每一行数据，在物理上都是寄生在这一张张 16KB 的磁盘页里的。

### 2. 在内存中：它是缓存的最小单元

因为磁盘的读写速度太慢了，MySQL 不可能每次查数据都去读一次硬盘。于是，MySQL 在内存中开辟了一块巨大的空间，叫做 **Buffer Pool（缓冲池）**。

这个 Buffer Pool 在内存里也被整整齐齐地切成了同样大小的块（也是 16KB），叫做**内存页（หรือ称 缓存页，Buffer Page）**。

### 它们是怎么联动起来的？（集装箱原理）

假设你要查询主键 `id = 1` 的数据：

1. **去内存找**：MySQL 首先看内存的 Buffer Pool 里有没有对应的内存页。

2. **磁盘加载**：如果内存里没有，MySQL **绝对不会**只去磁盘上把 `id = 1` 这一行（可能只有几百字节）抠出来。

3. **整页搬运**：MySQL 会把 `id = 1` 所在的这**整张 16KB 的磁盘页**，像搬运集装箱一样，原封不动地“拷贝”到内存的 Buffer Pool 中。

4. **内存读取**：数据到了内存页后，MySQL 再从这个内存页里把 `id = 1` 那一行的数据挑出来返回给你。

> 💡 **为什么非要整页搬运？（局部性原理）** 空间局部性原理告诉我们：如果你查了第 1 行数据，大概率你待会儿就要查第 2 行、第 3 行（比如你在做范围查询或者顺序读）。
> MySQL 直接把这 16KB 的一整页数据顺便都搬到内存里，下次你查邻近的数据时，直接命中内存，速度起飞。

# 原理流程

我们用一个**最底层的全流程拆解**，看看当你执行 `SELECT * FROM user WHERE id = 18` 时，MySQL 到底是怎么利用索引页找到数据的。

### 一、 索引在页里长什么样？

MySQL 的页分为很多种，最核心的是这两种：

1. **数据页（叶子节点）**：里面存的是**真正的整行数据**。

2. **索引页（非叶子节点）**：里面**不存**整行数据，只存“主键 ID + 下一层页的页号（Page Number）”。

假设我们现在有 3 个数据页（页号分别是 10, 11, 12），里面存着具体的数据行：

- **页 10**：存着主键 `id` 为 1 到 10 的数据。

- **页 11**：存着主键 `id` 为 11 到 20 的数据。

- **页 12**：存着主键 `id` 为 21 到 30 的数据。

为了能快速找到这三个数据页，MySQL 会建立一个**索引页（假设页号是 5）**。这个页 5 里面只存目录：

- `[id=1, 对应页号=10]`

- `[id=11, 对应页号=11]`

- `[id=21, 对应页号=12]`

这个**页 5**，就是 B+ 树的根节点。

### 二、 索引使用的全流程（以查找 id = 18 为例）

当你发送一句 `SELECT * FROM user WHERE id = 18;` 给 MySQL 时，真正的惊心动魄的寻找流程是这样的：

#### 第一步：定位根节点（根索引页）

MySQL 知道这张表的根节点是**页 5**。它首先去内存（Buffer Pool）里找页 5。

- *如果在内存中*：直接读取。

- *如果不在内存中*：去磁盘把页 5 搬到内存页里。

#### 第二步：在索引页中“做选择题”

MySQL 打开内存中的**页 5**（索引页），发现里面有三条目录：

- `id = 1` ➔ 往页 10 找

- `id = 11` ➔ 往页 11 找

- `id = 21` ➔ 往页 12 找

MySQL 一看，你要找的是 `18`。因为 $11 \le 18 < 21$，MySQL 瞬间做出判断：**目标绝对在页 11 里面！**

#### 第三步：加载目标数据页

MySQL 现在知道要去拿**页 11**了。它再次去内存（Buffer Pool）里看页 11 在不在。

- 假设此时页 11 不在内存，MySQL 发起一次 **磁盘 I/O**，把磁盘上的页 11（数据页）一整个（16KB）搬到内存页中。

#### 第四步：在数据页内部精确定位

现在**页 11**已经躺在内存里了。页 11 内部可能存了 10 行数据（id 从 11 到 20）。

- MySQL 页内部有一个叫做 **页目录（Page Directory）** 的槽位（Slot）设计，专门用来做**二分查找**。

- MySQL 在内存中用二分法，在页 11 内部瞬间定位到 `id = 18` 的那一行数据。

- 捞出这一行的数据，组装好返回给你。

# B+树与B树的区别

### 一、 核心区别对比

我们先来看直观的结构对比图：

#### 1. 数据存储的位置不同（最本质区别）

- **B 树**：**所有节点（不管是根节点、中间节点还是叶子节点）都既存索引键（Key），又存整行数据（Data）。**

- **B+ 树**：**非叶子节点只存索引键（Key）和指针，不存任何数据。** 所有的完整整行数据，都统一老老实实地挤在最底层的**叶子节点**里。

#### 2. 叶子节点的连接方式不同

- **B 树**：叶子节点之间是**独立的**，没有任何连接。

- **B+ 树**：所有的叶子节点之间，用一个**双向链表**全部串联了起来，像一条手拉手的传送带。

#### 3. 查询性能的稳定性不同

- **B 树**：查数据全凭运气。如果你要找的 `id=1` 刚好在根节点，那只要 1 次 I/O 就能返回；如果数据在叶子节点，就要找 3、4 次。

- **B+ 树**：因为非叶子节点没有数据，所有查询**必须一路走到最底层的叶子节点**才能拿到数据。所以它的查询耗时非常稳定，每一次都是固定的 2~3 次 I/O。

### 二、 为什么 MySQL 坚定地选择 B+ 树，而不是 B 树？

结合你之前学到的“页（16KB）”和“扇出”的概念，你可以从以下两个绝杀的工程角度去回答：

#### 理由一：B+ 树的“扇出”远大于 B 树（树更矮，I/O 更少）

- **在 B 树中**：因为每个节点都要存整行数据。假设一行数据 1KB，那么一个 16KB 的内存页，只能存 **16 条记录**。它的分叉（扇出）最多只有 16。为了存 2000 万条数据，B 树会变得**极高**（可能需要 5、6 层以上），查询时磁盘 I/O 频繁。

- **在 B+ 树中**：非叶子节点不存数据，只存主键（8字节）和指针（6字节）。一页可以存 **1170 个主键**，扇出极大。2000 万条数据只需要 3 层。

#### 理由二：B+ 树天然对“范围查询”友好，而 B 树极其痛苦

- 假设你要执行：`SELECT * FROM user WHERE id BETWEEN 10 AND 30;`

- **在 B+ 树里**：MySQL 先在树里用二分法找到 `id=10` 的叶子节点，然后不需要再爬树了，直接顺着底部的**双向链表**往右一路顺藤摸瓜，把 11、12...30 的数据页全拉出来。

- **在 B 树里**：它没有底部的链表。你找到了 10 之后，如果要找 11，你得先往上退回父节点，再往下走到右子节点（这叫中序遍历），要在树的上下层之间**反复横跳**。这会造成大量的、随机的磁盘 I/O，性能当场雪崩。

### 满分总结

> “B 树和 B+ 树最大的区别就在于**‘B+ 树的数据全在叶子节点，且叶子节点有双向链表’**。
> 
> 这个设计带来了两大优势：第一，因为非叶子节点不存数据，使得单个页能存更多索引，**极大地提高了扇出，让树变得更矮胖**，减少了磁盘 I/O 次数；第二，底部的双向链表让 **B+ 树在面对范围查询、排序和分页时，效率远远把 B 树甩在身后**。”

你现在闭上眼，对比一下那棵所有节点都臃肿地塞着数据的 B 树，和那棵清爽、只有底层有数据且手拉手的 B+ 树，它们的区别是不是完全清晰了？
