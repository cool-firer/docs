# Mysql VS PG

### 一、 核心架构：多进程 vs 多线程

这是它们最底层、最本质的区别。

- **MySQL (多线程模型)**：
  
  MySQL 为每个客户端连接创建一个线程（或从线程池中复用）。
  
  - **优点**：线程比进程更轻量，创建和销毁开销小，消耗的内存显著少于 PG，在大量并发连接时，内存管理更高效。
  
  - **缺点**：如果某个线程因为严重 Bug 导致崩溃（Crash），整个 MySQL 实例（所有连接）都会跟着一起崩溃。

- **PostgreSQL (多进程模型)**：
  
  PG 为每个客户端连接派生（Fork）一个独立的操作系统进程。
  
  - **优点**：隔离性极强。一个连接的进程挂了、内存溢出了，完全不影响其他进程，极为安全稳定。
  
  - **缺点**：进程非常消耗内存资源。在没有连接池（如 Pgbouncer）的情况下，PG 很难支撑像 MySQL 那么高并发的直接连接。

### 二、 并发控制：MVCC 的底层实现（核心高频考点）

虽然两者的隔离级别都基于 MVCC，但底层“存旧版本数据”的逻辑截然不同：

- **MySQL (Undo Log 机制)**：
  
  - **实现**：修改数据时，直接**原地修改**当前页的数据，把旧版本的数据写到单独的 `Undo Log`（回滚日志）里。
  
  - **影响**：表文件非常紧凑。即使大量更新，表也不会无限制膨胀，但需要依赖回滚段来读取老版本。

- **PostgreSQL (堆内多版本机制 / Append-Only)**：
  
  - **实现**：修改数据时，原数据不动，直接在表里**插入一行新数据**，通过隐藏的事务 ID（`xmin`, `xmax`）来标记谁能看。
  
  - **影响**：这就导致大量更新后，表里会充斥着大量的“死元组（废弃数据）”，必须依赖自带的 `VACUUM`（垃圾回收机制）来清理磁盘。如果清理不及时，会导致**表膨胀（Table Bloat）**，影响查询性能。

### 三、 功能扩展：复杂查询与数据类型

- **MySQL**：
  
  - 过去比较弱，虽然现在（MySQL 8.0+）支持了窗口函数、CTE（通用表表达式）和 JSON，但总体上还是更偏向于简单、高频、快速的 OLTP（联机事务处理）业务。

- **PostgreSQL（功能完胜）**：
  
  - **支持的数据类型极多**：原生支持数组（Array）、多维几何数据、网络地址（CIDR）等。
  
  - **地理信息系统 (GIS)**：PG 拥有杀手级插件 **PostGIS**，是目前全世界处理地理位置、地图、经纬度数据最强大的数据库，这一点 MySQL 望尘莫及。
  
  - **可扩展性**：PG 支持“插件（Extension）”机制（如时序数据插件 TimescaleDB、向量检索插件 pgvector 等），甚至允许你用 Python 或 Partman 写存储过程。

### 四、 索引支持的区别

虽然两者都以 B+ 树（PG 里叫 B-Tree，但实现类似）为主，但 PG 提供的索引大军要丰富得多：

- **MySQL**：主要就是 B+ 树索引，外加少量的全文索引（Full-Text）和空间索引。

- **PostgreSQL**：除了 B-Tree，还原生支持 **GIN 索引**（专门用来给 JSON、数组做倒排索引，查 JSON 快到飞起）、**BRIN 索引**（块范围索引，适合按时间排序的超大数据集）、**GiST/SP-GiST 索引**（适合地理位置和范围数据）。

### 📊 总结：企业如何选型？

在面试或实际架构选型中，你可以这样总结它们的适用场景：

| **维度**    | **MySQL**                       | **PostgreSQL**                            |
| --------- | ------------------------------- | ----------------------------------------- |
| **设计核心**  | 简单、高效、快速响应                      | 严谨、功能强大、高度可扩展                             |
| **高并发读写** | 更好（多线程、内存占用低）                   | 需要连接池辅助，但处理复杂SQL能力更强                      |
| **数据安全性** | 很好，但不及 PG 严格                    | 极好，全面遵循 ANSI SQL 标准，杜绝数据隐式转换              |
| **推荐场景**  | 互联网、电商、传统 Web 应用（多读少写、业务逻辑在代码层） | 地理信息（GIS）、大宽表分析、财务金融、时序数据、需要存储复杂 JSON 的业务 |

> 💡 **一句话绝杀口诀：**
> 
> 如果你的业务是**典型的互联网应用**，追求高并发、高可用，且查询大多比较简单，选 **MySQL**。
> 
> 如果你的业务涉及**复杂的报表统计、地理位置信息（GIS）、或者是高频的 JSON 科学计算**，选 **PostgreSQL**。
