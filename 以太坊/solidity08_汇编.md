# 内联汇编 (Inline Assembly)

如果说 Solidity 是高级语言（像 C++ 或 Java），那么内联汇编就是底层的汇编语言，它让你直接与以太坊虚拟机（EVM）的寄存器和内存打交道。Solidity 中使用的内联汇编语言被称为 **Yul**。

内联汇编是低级访问 EVM 的方式。它**绕过了 Solidity 的许多重要安全特性和检查**。只有在确实需要且你对使用它充满信心时才应使用。

- **标记**：内联汇编块由 `assembly { ... }` 标记。

- **访问变量**：内联汇编可以直接访问外部 Solidity 定义的局部变量（下文会详述）。

- **独立命名空间**：不同的 `assembly` 块之间不共享命名空间。你不能在一个块里调用另一个块定义的 Yul 函数或变量。

Solidity 帮我们处理了很多麻烦事，但在汇编里，你必须自己动手：

1. **没有溢出检查**：在 Solidity 0.8+ 中，`a + b` 会自动检查溢出；而在汇编里，`add(a, b)` 发生溢出就直接回绕（Wrap around），不会报错。

2. **内存管理**：Solidity 有一个“空闲内存指针”来管理内存。如果你在汇编里乱写内存（`mstore`），可能会覆盖掉 Solidity 正在使用的重要数据，导致程序崩溃或出现逻辑漏洞。

3. **类型检查失效**：汇编眼中只有 `uint256`（32字节的字）。如果你把一个 `address` 当成 `uint` 来算，汇编不会拦着你。

```solidity
function getLength(bytes memory data) public pure returns (uint len) {
    assembly {
        // 在 Solidity 中，动态字节数组的第一个字（32字节）存储的是长度
        // mload 表示“从内存加载”
        // data 变量本身在内存中其实是一个指向起始位置的指针
        len := mload(data)
    }
}
```

# 访问外部变量、函数和库

你可以直接在汇编中使用 Solidity 变量名：

- **值类型 (Value Types)**：可以直接读取或赋值。

- **内存变量 (Memory Variables)**：变量名代表的是该变量在**内存中的地址**（指针），而不是数据本身。赋值只会改变指针指向，不改变原数据。

- **Calldata 变量**：
  
  - **静态大小**：代表数据在 calldata 中的地址。
  
  - **动态大小**（如 `bytes`, `uint[]`）：使用 `x.offset` 获取起始位置偏移量，使用 `x.length` 获取元素数量。

- **存储变量 (Storage Variables)**：由于存储槽可能被多个变量共享（紧凑打包），需要两个值来定位：
  
  - `x.slot`：变量所在的存储槽编号。
  
  - `x.offset`：变量在该槽位内的字节偏移量。
  
  - *注意：直接使用 `x` 会报错。*

- **函数指针**：外部函数指针可以使用 `x.address` 和 `x.selector` 访问。

**警告（数据清洗）**：

对于小于 256 位的类型（如 `uint32`, `address`），汇编读取时**高位可能包含脏数据**（即不一定是 0）。在使用前必须手动清洗（例如使用 `and` 掩码）。

---

深度解析：不同变量在汇编中的“真面目”

这是最容易让初学者困惑的地方。在汇编块 `{ ... }` 里，当你写下一个变量名时，它到底代表什么？

| **变量类型**                    | **汇编中变量名的含义**      | **辅助属性**               |
| --------------------------- | ------------------ | ---------------------- |
| `uint256`, `bool`           | **值本身**            | 无                      |
| `string`, `struct` (memory) | **内存地址 (Pointer)** | 无                      |
| `uint[]` (calldata)         | **报错** (需使用属性)     | `x.offset`, `x.length` |
| `uint256` (storage)         | **报错** (需使用属性)     | `x.slot`, `x.offset`   |

---

 代码示例与注释

A. 内存指针的操作

```solidity
function pointerDemo() public pure returns (uint r) {
    uint[] memory arr = new uint[](3);
    arr[0] = 50;

    assembly {
        // arr 此时是一个内存地址
        // mload(arr) 获取的是该地址存储的第一个字，即数组长度
        let length := mload(arr) 

        // 要获取 arr[0]，需要跳过第一个字的长度字段（32字节）
        r := mload(add(arr, 32)) // r = 50
    }
}
```

B. 存储变量的精准打击

由于 Solidity 会把小的变量挤在一个槽里，你必须配合 `slot` 和 `offset` 使用。

```solidity
contract StorageTest {
    uint128 public a = 1;
    uint128 public b = 2; // a 和 b 共享槽位 0

    function getB() public view returns (uint128 res) {
        assembly {
            // b.slot 是 0
            // b.offset 是 16 (因为 a 占了前 16 字节)
            let v := sload(b.slot)
            // 将 v 右移 16 字节并截断，才能拿到真正的 b
            res := shr(mul(b.offset, 8), v)
        }
    }
}
```

# 内存管理 (Memory Management)

Solidity 内存布局有一套严格的“领土划分”：

1. **空闲内存指针 (Free Memory Pointer)**：位于内存 `0x40` 位置。它存储的值指向当前未被使用的内存起始地址。

2. **分配逻辑**：分配内存需从 `mload(0x40)` 开始，并更新 `0x40` 处的值（增加分配长度）。

3. **内存区域划分**：
   
   - `0x00 - 0x3f` (64字节)：**暂存空间 (Scratch space)**，用于哈希运算等短期任务。
   
   - `0x40 - 0x5f` (32字节)：**空闲内存指针**。
   
   - `0x60 - 0x7f` (32字节)：**零值插槽 (Zero slot)**，永远保持为 0，用作空动态数组的初始值。
   
   - `0x80` 及以后：**可分配内存**。

**数组在内存中的结构：**

- **动态数组**：第一个插槽（32字节）存储**长度**，随后是元素。

- **内存对齐**：除了 `bytes` 和 `string`，内存数组中的元素永远占据 32 字节的倍数（即便类型是 `bytes1`）。

---

## 深度解析：内存的“四大天王”

Solidity 的内存布局像是一张规划好的城市地图，你绝对不能在“绿化带”上盖房子。

### A. 为什么 0x40 这么重要？

在汇编里，如果你直接 `mstore(0x80, value)`，你可能会覆盖掉 Solidity 正在使用的变量。 **正确做法：**

Solidity

```
assembly {
    let ptr := mload(0x40)   // 问 Solidity：现在哪块地是空的？
    mstore(ptr, 0x123)       // 在空地上存东西
    mstore(0x40, add(ptr, 32)) // 告诉 Solidity：这块地我用了，下一块地从后面开始
}
```

### B. 0x60 的“幽灵插槽”

如果你声明一个 `uint[] memory a;` 但不初始化它，它的指针默认指向 `0x60`。因为 `0x60` 处的值永远是 0，所以当你读取 `a.length` 时，会正好读到 `0x60` 里的 0。

---

## 数组的内存真相 (Memory Array Layout)

这是一个经常让开发者掉坑的地方：**内存中的 `uint8[]` 并不省空间。**

```solidity
uint8[] memory smallArray = new uint8[](2);
```

在内存（Memory）中，它的布局如下：

1. **位置 P**：`0x000...002` (长度为 2)

2. **位置 P + 32字节**：`0x000...0x01` (第一个元素)

3. **位置 P + 64字节**：`0x000...0x02` (第二个元素)

**注意：** 每个 `uint8` 依然占据了 **32 字节** 的空间。这是为了方便 EVM 访问（EVM 每次读写固定是 32 字节）。

- *例外*：`bytes` 和 `string` 会进行紧凑排列，不会每个字节占 32 字节。

---

## 警告：静态数组的长度

```solidity
uint[2] memory staticArray;
```

目前的 Solidity 版本中，`staticArray` 在内存里**没有**长度字段。如果你用汇编去读它的第一个字，读到的是第一个元素的值，而不是长度。文档提示未来可能会变，所以**不要在汇编里依赖“静态数组没有长度”这个特性**。

# 内存安全 (Memory Safety)

**内存安全 (Memory Safety)**

- **冲突点**：编译器默认内存是井然有序的。但内联汇编可以不遵守规则。因此，一旦代码中出现包含内存操作的 `assembly` 块，编译器为了安全，会**全局禁用**某些关键优化。

- **解决方案**：如果你确信你的汇编代码遵守了 Solidity 的内存模型，你可以给它贴上 `("memory-safe")` 的标签。

- **内存安全的准则**：一个标记为内存安全的汇编块，**只能**访问以下范围：
  
  1. 你通过更新空闲内存指针（`0x40`）手动分配的内存。
  
  2. Solidity 分配的内存（如你引用的内存数组范围内的内存）。
  
  3. `0x00 - 0x3f` 的 **Scratch Space（暂存空间）**。
  
  4. **临时空间**：即当前空闲内存指针（Free Memory Pointer）**之后**的区域，且你不需要更新指针（前提是后续没有其他的内存分配）。

**警告：** 如果你标记了 `memory-safe` 却违反了上述规则（例如写到了 `0x60` 或者是未分配的区域），编译器生成的优化代码可能会与你的汇编代码产生冲突，导致**完全不可预知且难以通过测试发现的错误**。

---

## 深度解析：什么是“不安全”的操作？

文档中举了一个非常经典的例子：错误示例：覆盖了敏感区域

```solidity
assembly {
  // 把返回数据拷贝到内存 0 地址开始的地方
  // 如果返回数据超过 64 字节，它会覆盖 0x40 位置的空闲内存指针！
  // 这会彻底破坏 Solidity 后续的内存分配逻辑。
  returndatacopy(0, 0, returndatasize())
  revert(0, returndatasize())
}
```

这段代码**不安全**，因为它可能污染了 `0x40`（指针）或 `0x60`（零值插槽）。

正确示例：尊重指针

```solidity
assembly ("memory-safe") {
  let p := mload(0x40) // 先看当前空闲地块在哪
  returndatacopy(p, 0, returndatasize()) // 从空闲地块开始写
  revert(p, returndatasize())
}
```

这段代码是**安全**的。即使你没有更新 `0x40` 的值也没关系，因为 `revert` 之后合约就停止执行了。

---

## 内存布局可视化

为了理解为什么要遵守这些范围，我们需要看清楚内存的“雷区”：

- **0x00 - 0x3F**: 安全。你可以随便折腾。

- **0x40 - 0x5F**: **极度危险**。这是指针，一旦改错，整个合约的内存管理就瘫痪了。

- **0x60 - 0x7F**: **危险**。这是空数组的标识，改了它，你之后创建的所有空数组都会带上“脏数据”。

- **0x80+**: **按需使用**。必须配合 `0x40` 处的指针来使用。

---

## 为什么要费力打这个标签？

你可能会问：“我不打这个标签，合约不是也能跑吗？”

**答案是：Gas 和 栈空间。**

1. **解决 Stack Too Deep**：当你的函数里局部变量太多时，编译器需要把一部分变量存到内存里。如果不打 `memory-safe` 标签，编译器就不敢这么做，你的代码可能直接编不过。

2. **高性能优化**：Yul IR 优化器可以跨步骤移动内存操作，从而大幅节省 Gas。只有你承诺“我不乱动内存”，它才敢放开手脚去优化。

---

## 总结与最佳实践

1. **默认保守**：如果你对内存布局不是 100% 确定，不要加 `("memory-safe")`。

2. **临时使用**：如果你只是想临时存一点数据做哈希，首选 `0x00 - 0x3f`（Scratch Space）。

3. **大宗数据**：永远先从 `0x40` 读取指针，并在使用后视情况更新它。

4. **避免指针欺骗**：永远不要手动给一个 Solidity 引用变量（如 `bytes memory x`）赋一个魔数地址（如 `x := 0x40`），这在任何时候都是极不安全的。
