# 清洗变量 (Cleaning Up Variables)

在 EVM 的世界里，所有的“容器”（栈、内存、存储）标准尺寸都是 **256 位**。如果你使用的是 `uint8` 这样的小尺寸类型，那么剩下的 248 位该怎么办？这就是 **变量清洗（Cleaning Up Variables）** 要解决的问题。

- **原则**：当一个值的位宽小于 256 位时，剩余的高位必须被清理。Solidity 编译器会在任何可能被“高位垃圾数据”影响的操作之前执行清洗。

- **为什么要清洗？**：
  
  - **写入内存前**：因为内存内容可能被用于计算哈希（哈希对每一位都很敏感）或作为消息调用的数据发送。
  
  - **存入存储前**：防止观察到被污染的脏数据。

- **什么时候不清洗？**：
  
  - 如果紧接着的操作不受高位影响。例如，`JUMPI` 指令将任何非零值都视为 `true`，所以布尔值在做条件跳转前不需要清洗。
  
  - **内联汇编**：如果你用汇编访问小于 256 位的变量，编译器**不保证**这些值是干净的。



## 深度解析：符号扩展 vs 零填充

这是初学者最容易在底层调试中混淆的地方。

A. 无符号整数 (uint8 示例)

如果你有一个 `uint8`，它的有效位只有最后 8 位。

- **脏数据**：`0101...1010`（高位有乱码）。

- **清洗后**：`0000...1010`（高位全部强制抹零）。

B. 有符号整数 (int8 示例)

有符号数的清洗涉及到“符号位”：

- **正数**：如果第 8 位是 `0`，那么高位全部补 `0`。

- **负数**：如果第 8 位是 `1`（代表负数），那么高位全部补 `1`（即 `0xfff...`）。
  
  - *例如：`int8` 的 `-1` 补码是 `11111111`。在 256 位下，它必须表现为全 `1`，否则它的数学意义就变了。*

## 开发者的避坑指南 ⚠️

### 1. 永远不要在汇编里假设变量是干净的

这是文档最直接的警告。

```solidity
uint8 a = 1;
assembly {
    // 此时栈上的 a 可能是 0x00...0001，
    // 但也可能是 0xabc...0001
    let x := a 
}
```

如果你在汇编里直接用这个 `x` 去参与哈希计算，结果可能每次都不一样，导致严重的 Bug。**在汇编里使用小类型变量前，手动执行 `and(a, 0xff)` 是一种安全的习惯。**

### 2. 布尔值的特殊性

在 Solidity 逻辑层，`bool` 只有 `true` 和 `false`。但在 EVM 层，它是 `1` 和 `0`。

有趣的是，如果你通过汇编强行把一个 `bool` 变量设为 `2`：

- 在 Solidity 的 `if` 判断里，它能通过（因为非零）。

- 但如果你把它传给另一个合约，或者存入存储，它会被编译器强制清洗回 `1`。
