#内存中的“四大金刚”：保留插槽

当你启动一个合约调用时，内存的前 **128 字节**（4 个 32 字节插槽）是预留给编译器特殊用途的：

1. **0x00 - 0x3f (64 字节)：暂存空间 (Scratch Space)**
   
   - 主要用于哈希运算（比如 `keccak256`）。
   
   - 在两条语句之间（如内联汇编里），你可以随意使用它。

2. **0x40 - 0x5f (32 字节)：自由内存指针 (Free Memory Pointer)**
   
   - 这是内存的“路标”。它存储了一个数值，告诉编译器：**“当前可用的空闲内存从哪个地址开始？”**。
   
   - 初始值通常是 `0x80`（即 128 字节处）。

3. **0x60 - 0x7f (32 字节)：零插槽 (Zero Slot)**
   
   - 用作动态内存数组的初始值（空引用）。
   
   - **千万不能往这里写数据！** 否则会破坏程序的逻辑判定。

# 内存分配：只增不减

Solidity 处理内存的方式非常简单（甚至有点原始）：

- **分配逻辑**：每当你创建一个新的内存对象（变量、数组），它就直接放在“自由内存指针”指向的位置，然后把指针向后推。

- **不释放机制**：**内存永远不会被释放（Free）**。在一次交易执行过程中，内存只会越用越多。虽然未来可能会引入垃圾回收，但目前就是这么霸气。



# 核心区别：不再“紧凑”

这是最让开发者迷惑的地方：**内存中没有紧凑打包（Packing）！**

## A. 数组的区别

- **Storage 中**：`uint8[4]` 只需要 **32 字节**（4 个 1 字节变量挤在一起）。

- **Memory 中**：`uint8[4]` 需要 **128 字节**（每个元素强行占据 32 字节）。

- **原因**：EVM 处理 32 字节数据的速度最快。内存是临时性的，Gas 消耗相对低，所以编译器优先选择“速度”而非“空间”。

## B. 结构体的区别

```solidity
struct S {
    uint a;   // 32字节
    uint b;   // 32字节
    uint8 c;  // 1字节
    uint8 d;  // 1字节
}
```

- **Storage**: 占用 **3 个 Slot**（`a` 一个，`b` 一个，`c` 和 `d` 挤在一个）。

- **Memory**: 占用 **4 个 Slot**（`a`, `b`, `c`, `d` 各占一个完整的 32 字节）。

## 动态内存数组的结构

内存中的动态数组（如 `uint[] memory a`）存储非常直观：

1. **第一个插槽**：存储数组的**长度**。

2. **后续插槽**：存储数组的**元素**（每个元素 32 字节）。
- *注意：多维数组在内存中其实就是一堆“指向其他数组地址的指针”。*



# 危险警告：非零内存

文档里提到了一个非常隐蔽的坑：
有些操作（如某些系统级的拷贝）会临时借用“自由内存指针”指向的区域，但它们**用完后不会更新指针**。

- **后果**：当你正式申请内存时，拿到的那块地可能已经被之前的临时操作“弄脏”了（里面有残余数据）。

- **教训**：永远不要假设新分配的内存是全零的。
