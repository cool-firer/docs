# Operators运算符

## **两个变量a, b运算结果的类型规则**

- 如果右操作数的类型可以隐式转换为左操作数的类型，则使用左操作数的类型。

- 如果左操作数的类型可以隐式转换为右操作数的类型，则使用右操作数的类型。

- 否则，该运算是不允许的。核心逻辑：向“大”看齐
  
  ```solidity
  uint8 + uint32
  1.先检查：uint32 -> uint8 不允许。因为可能截断。
  2.再检查：uint8 -> uint32 允许，所以最终用 uint32 运算。
  
  int8 + uint8
  不允许。因为两者转换都不安全。
  ```

## 其中一个是字面量时

把字面量当成可以容纳这个字面量的最小类型。同位宽下，unsigned 类型比 signed 类型更“小”。

```solidity
1 默认不是uint256，而是uint8
255最小uint8
256最小uint16


-1最小
```

前面文档讲的，精度无限制。



## 例外情况

这部分是开发者最容易踩坑的地方：

- **`x << y` 或 `x y`：** 结果的类型**完全由 `x` 决定**。

- **风险：** 即使 `y` 是一个巨大的 `uint256`，如果 `x` 是 `uint8`，运算依然在 8 位空间内完成。如果结果超过 255，就会发生溢出报错或截断。



## Ternary Operator三元运算符

三元加上字面量有点小坑点：

```solidity
三元表达式本身也有类型，例如：(true ? 1 : 0)，而是一个uint8。


所以：255 + (true ? 1 : 0)会overflow。


1.5 + (true ? 1.5 : 2.5)
三元在类型，但Solidity 没有浮点型，不允许转换小数 --> 整数。
因此编译失败。
```



## delete

**它是“赋值”，不是“销毁”**

在以太坊虚拟机（EVM）中，存储数据是需要花钱（Gas）的，而将数据清零可以获得 Gas 返还。因此，`delete` 本质上是将存储槽（Storage Slot）重新设置为 `0x0`。它不会像 Python 或 Java 那样有垃圾回收机制，它只是把值改回默认状态。

`delete` 操作结果全对比表

| **类型**             | **操作**          | **delete 后的效果**          | **对长度/结构的影响**       |
| ------------------ | --------------- | ------------------------ | ------------------- |
| **动态数组 `T[] a`**   | `delete a`      | 数组重置为**空**               | `a.length` 变为 **0** |
| **动态数组 `T[] a`**   | `delete a[x]`   | 索引 `x` 处的值变为 **0/初始值**   | **长度不变**，留下一个“空洞”   |
| **静态数组 `T[n] a`**  | `delete a`      | **所有**元素变为初始值            | 长度 `n` 保持不变         |
| **静态数组 `T[n] a`**  | `delete a[x]`   | 索引 `x` 处的值变为初始值          | 长度 `n` 保持不变         |
| **映射 `mapping a`** | `delete a`      | **不允许操作 (报错)**           | 无法一次性重置整个 Mapping   |
| **映射 `mapping a`** | `delete a[key]` | 该 `key` 对应的值变为 **0/初始值** | 逻辑上相当于删除了该键值对       |
| **结构体 `Struct a`** | `delete a`      | 所有非 mapping 成员**递归重置**   | 结构体依然存在，但内容清空       |

# 基础类型之间的转换

可以理解为变量这间的转换。两个都是变量。

## 隐式转换Implicit Convertions

编译器会在赋值、向函数传递参数以及使用操作符时，自动执行隐式类型转换。通常情况下，如果值类型之间的转换在**语义上是合理的**且**没有信息丢失**，那么这种隐式转换就是可行的。

例如：`uint8` 可以转换为 `uint16`，`int128` 可以转换为 `int256`。但是 `int8` **不能**转换为 `uint256`，因为 `uint256` 无法保存像 `-1` 这样的负数值。

如果对不同类型的操作数使用操作符，编译器会尝试将其中一个操作数隐式转换为另一个操作数的类型（赋值时也是如此）。这意味着**运算始终在其中一个操作数的类型空间内执行**。



## 显式转换Explicit Convertions

如果编译器不允许隐式转换，但你确信转换可以正常工作，那么有时可以使用显式类型转换。

将负数 `int` 转换为 `uint` 的示例：

```solidity
int  y = -3;
uint x = uint(y);
```

显式转换有四种不同的情况。

### 1. 整数转换代码示例 (Integer)

整数转换优先保护“个位”数值，就像我们在纸上写数字总是右对齐一样。

```solidity
// 情况 A：向小类型转（截断）
uint32 a = 0x12345678;
uint16 b = uint16(a); 
// 结果：0x5678 （左边的 1234 被砍掉了）

// 情况 B：向大类型转（填充）
uint16 c = 0x1234;
uint32 d = uint32(c);
// 结果：0x00001234 （左边补了 4 个零，数值大小依然是 0x1234）
```

---

### 2. 定长字节数组示例 (Fixed Bytes)

字节数组像是一排货架上的商品，从索引 0 开始排，所以转换时优先保护“开头”。

```solidity
// 情况 A：向小类型转（截断）
bytes2 a = 0x1234;
bytes1 b = bytes1(a);
// 结果：0x12 （右边的 34 被砍掉了，保留了开头的第 0 位）

// 情况 B：向大类型转（填充）
bytes2 c = 0x1234;
bytes4 d = bytes4(c);
// 结果：0x12340000 （右边补零，确保 d[0] 依然是 0x12）
```

---

### 3. 跨种类转换（必须中转）

如果你尝试直接把 `bytes2` 转成 `uint32`，编译器会因为“填充方向冲突”而报错。你必须通过一个**中间人**来明确你的意图。

**你想保值（右对齐）还是保序（左对齐）？**

```solidity
bytes2 val = 0x1234;

// 意图 1：我想把它当成数字 0x1234 处理
// 步骤：先转成同宽度的 uint16，再扩充到 uint32
uint32 asNumber = uint32(uint16(val)); 
// 结果：0x00001234

// 意图 2：我想保持字节序列的开头不变
// 步骤：先扩充到 bytes4，再转成同宽度的 uint32
uint32 asSequence = uint32(bytes4(val)); 
// 结果：0x12340000
```

---

### 4. 动态字节数组转定长 (Bytes to BytesN)

这是 Solidity 0.8.x 之后非常实用的特性：

```solidity
bytes memory data = "abcdefgh"; // 8字节

// 转成更短的
bytes3 short = bytes3(data); 
// 结果："abc" (截断末尾)

// 转成更长的
bytes10 long = bytes10(data);
// 结果："abcdefgh\0\0" (末尾补零)
```

# 字面量与基础类型之间的转换

## 整数0.8.0 之后的变化

这一段的核心在于：**字面量不再允许通过“强转”来强制截断。**

**A. 隐式转换：安全边界**

当你写 `uint8 a = 12;` 时，编译器会检查 `12` 是否在 `uint8` 的范围（0 到 255）内。

- 如果是 `uint8 a = 300;`，编译器直接报错。这很好理解，因为“小盒子装不下大东西”。

**B. 显式转换：规则的收紧（重点）**

在旧版本（< 0.8.0）中，如果你想强行把一个大数塞进小变量，你可以“骗”编译器：

- **旧版写法：** `uint8 a = uint8(300);`
  
  - 在旧版里这是**允许**的。结果 `a` 会变成 `44`（即 $300 \pmod{256}$）。这往往会导致难以察觉的计算逻辑 Bug。

- **新版（0.8.0+）写法：** `uint8 a = uint8(300);`
  
  - 在现在的版本中，这会**直接编译报错**。编译器会告诉你：`300` 超出了 `uint8` 的范围。

## 定长字节数组Fixed-Size

**十进制字面量**不能隐式转换为定长字节数组。**十六进制字面量**可以转换，但前提是：十六进制数字的位数（digits）必须**精确匹配**字节类型的长度。作为一个例外，值为零的十进制和十六进制字面量都可以转换为任何定长字节数组类型：

Solidity

```
bytes2 a = 54321;  // 不允许（十进制不能直接转 bytes）
bytes2 b = 0x12;   // 不允许（位数不匹配：2位 vs 4位）
bytes2 c = 0x123;  // 不允许（位数不匹配：3位 vs 4位）
bytes2 d = 0x1234; // 正常（4位十六进制正好是 2 字节）
bytes2 e = 0x0012; // 正常（通过补零达到 4 位，匹配 2 字节）
bytes4 f = 0;      // 正常（例外：零值可以）
bytes4 g = 0x0;    // 正常（例外：零值可以）
```

**字符串字面量**和**十六进制字符串字面量**可以隐式转换为定长字节数组，前提是它们的字符数（或字节数）**小于或等于**字节类型的长度：

Solidity

```
bytes2 a = hex"1234"; // 正常（正好 2 字节）
bytes2 b = "xy";      // 正常（正好 2 字节）
bytes2 c = hex"12";   // 正常（1 字节 < 2 字节，右侧补零）
bytes2 e = "x";       // 正常（1 字节 < 2 字节，右侧补零）
bytes2 f = "xyz";     // 不允许（3 字节 > 2 字节，放不下）
```

## 地址Address

地址转换规则分为以下四类：

1. 相同长度类型间的显式转换 (核心规则)

以太坊地址是 160 位（20字节）。因此，只有同样是 160 位的数据类型可以**直接**显式转换为 `address`。

- **允许类型：** `uint160` 和 `bytes20`。

- **转换方式：** `address(x)`。

Solidity

```
uint160 n = 12345;
bytes20 b = 0x1234567890123456789012345678901234567890;

address addr1 = address(n); // 正确
address addr2 = address(b); // 正确
```

---

2. 不同长度整数的转换 (间接转换)

如果你有一个 `uint256`（最常用的类型）或 `uint8`，你**不能**直接转为 `address`。必须先中转为 `uint160`。

- **规则：** `uint(不同长度)` $\rightarrow$ `uint160` $\rightarrow$ `address`。

Solidity

```
uint256 bigNum = 0x123...abc;
uint8 smallNum = 10;

// address a = address(bigNum);     // 报错！
address a = address(uint160(bigNum)); // 正确：先截断高位再转换

// address b = address(smallNum);   // 报错！
address b = address(uint160(smallNum)); // 正确：先填充高位零再转换
```

---

3. 普通地址与可支付地址的转换 (`payable`)

这是智能合约开发中最常见的转换。

- **`address` 转 `address payable`：** 使用 `payable(addr)`。

- **`address payable` 转 `address`：** 隐式转换（自动允许，因为普通地址权限更小）。

Solidity

```
address addr = 0x123...;
address payable pAddr = payable(addr); // 显式转换，使其可以接收ETH

address addrAgain = pAddr; // 隐式转换，直接赋值即可
```

---

4. 合约对象与地址的转换

合约实例本质上就是一个地址。

- **合约转地址：** `address(this)` 或 `address(contractInstance)`。

- **地址转合约：** `ContractName(addr)`。

Solidity

```
contract MyContract {
    function getInfo() public view {
        address myAddr = address(this); // 获取当前合约地址
    }
}

// 在另一个合约中
address addr = 0xABC...;
MyContract instance = MyContract(addr); // 将地址声明为合约对象
```
