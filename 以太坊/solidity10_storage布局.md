# 状态变量State Variables

## 状态变量在存储与瞬态存储中的布局

- **基本原则**：存储是连续的，从插槽 0（Slot 0）开始。除了动态数组和映射外，数据按顺序存放。

- **紧凑存储（Packing）**：多个小于 32 字节的变量会尽可能挤在一个插槽里。
  
  - 插槽内的第一个元素是**低位对齐**的。
  
  - 如果当前插槽放不下某个变量，它会跳到下一个插槽起始位置。
  
  - **结构体（Struct）和数组**：永远从一个**新插槽**开始，其内部元素遵循同样的紧凑规则。
  
  - 结构体或数组之后的变量，也必须从**新插槽**开始。

- **继承规则**：按 C3 线性化顺序（从最基类到最派生类）排列。不同合约的变量**可以**共享同一个插槽。

- **自定义偏移 (`layout at X`)**：这是新特性。你可以指定存储从特定的插槽编号开始（例如 Slot 42）。

- **瞬态存储 (`transient`)**：
  
  - 仅支持值类型。
  
  - 布局与普通存储完全独立，互不干扰。

- **常量与不可变量 (`constant`/`immutable`)**：不占用存储空间，不影响布局。

## 完整代码结构分析

```solidity
// SPDX-License-Identifier: GPL-3.0
pragma solidity ^0.8.29;

struct S {
    int32 x;  // 4 字节
    bool y;   /* 1 字节 */
}

contract A {
    uint a;                   // 32 字节 (uint256)
    uint128 transient b;      // 16 字节 (瞬态存储，不占常规 Slot)
    uint constant c = 10;     // 常量 (内联在代码中，不占 Slot)
    uint immutable d = 12;    // 不可变量 (部署时嵌入字节码，不占 Slot)
}

contract B {
    uint8[] e;                // 动态数组 (占用 1 个 Slot 存长度/指引)
    mapping(uint => S) f;     // 映射 (占用 1 个 Slot 作为哈希计算的基准)
    uint16 g;                 // 2 字节
    uint16 h;                 // 2 字节
    bytes16 transient i;      // 16 字节 (瞬态存储)
    S s;                      // 结构体 (内含 5 字节，但会另起新槽)
    int8 k;                   // 1 字节
}

// 核心：C 继承 A 和 B，并强制从 Slot 42 开始存储
contract C is A, B layout at 42 {
    bytes21 l;                // 21 字节
    uint8[10] m;              // 静态数组 (10 字节)
    bytes5[8] n;              // 静态数组 (5 * 8 = 40 字节，跨越 2 个 Slot)
    bytes5 o;                 // 5 字节
}
```

## 存储布局逐行拆解 (Storage Layout)

Solidity 按照 **C3 线性化顺序**（从最基类到子类）进行排布。在这里顺序是：`A` -> `B` -> `C`。由于设置了 `layout at 42`，所有原本从 0 开始的编号全部加 42。

第一阶段：来自 A 的变量

- **Slot 42**: 变量 `a` (uint256)。
  
  - 它占据了完整的 32 字节。
  
  - *注意：`b`, `c`, `d` 因为分别是 transient, constant, immutable，全部跳过。*

第二阶段：来自 B 的变量

- **Slot 43**: 变量 `e` (uint8[])。
  
  - 动态类型总是独自占据一个完整的 Slot。

- **Slot 44**: 变量 `f` (mapping)。
  
  - 映射总是独自占据一个完整的 Slot。

- **Slot 45**: 变量 `g` (uint16) 和 `h` (uint16)。
  
  - `g` 占据偏移量 0-1 字节，`h` 占据 2-3 字节。
  
  - 这个槽位还剩 28 字节，但下一个变量是结构体。

- **Slot 46**: 变量 `s` (结构体 S)。
  
  - **规则**：结构体必须从新插槽开始。
  
  - 内含 `x` (4字节) 和 `y` (1字节)，共占 5 字节。

- **Slot 47**: 变量 `k` (int8) 和来自 C 的 `l` (bytes21)。
  
  - **规则**：结构体之后的变量必须另起新槽。
  
  - `k` 占 1 字节。此时 `l` 只有 21 字节，槽位空余足够，因此 `l` 紧跟在 `k` 后面。

第三阶段：来自 C 的变量

- **Slot 48**: 变量 `m` (uint8[10])。
  
  - 静态数组另起新槽，占 10 字节。

- **Slot 49 & 50**: 变量 `n` (bytes5[8])。
  
  - 总大小 40 字节。
  
  - **Slot 49** 存下前 6 个元素（30字节），最后 2 字节放不下第 7 个元素。
  
  - **Slot 50** 存下剩下的 2 个元素（10字节）。

- **Slot 51**: 变量 `o` (bytes5)。
  
  - **规则**：数组之后的变量另起新槽。

## 瞬态存储布局 (Transient Storage)

瞬态存储的 Slot 计数器是**独立**的，且不受 `layout at 42` 的影响（因为它不属于永久存储）。

- **Slot 00**:
  
  - 首先是 `B` 中的 `i` (bytes16)，占据 0-15 字节。
  
  - 然后是 `A` 中的 `b` (uint128)，占据 16-31 字节。
  
  - 它们完美地挤在了一个 32 字节的插槽里。

## 关键点深度总结

1. **打包逻辑 (Packing)**：
   Solidity 会尝试节省空间，但有三个“硬边界”会强制开启新槽位：
   
   - **结构体 (Struct)** 的开始。
   
   - **静态数组 (Static Array)** 的开始。
   
   - **结构体或数组结束后的第一个变量**。

2. **继承间的打包**：
   父类末尾的空隙**可以**被子类的变量填补（如上述 Slot 47 中 `k` 和 `l` 的合并），前提是子类变量不是结构体或数组。

3. **为什么 `layout at 42` 很重要？**：
   在复杂的代理合约（Proxy）架构中，为了防止不同逻辑合约之间的存储冲突（Storage Collision），开发者通常需要手动指定存储基准地址。以前这需要通过汇编或极其复杂的占位变量来实现，现在可以直接通过 `layout at` 语法糖解决。

4. **性能警告**：
   虽然打包变量省了空间（SSTORE 变便宜了），但如果你在一段代码里只操作 `g` 而不碰 `h`，EVM 依然要读取整个 Slot 45。如果频繁修改打包在一起的变量中的某一个，有时反而会比分开存更贵。

# 寻址

## 动态数组 (Dynamic Arrays)

- **占位符 (Slot $p$)**：存储数组的**长度**。

- **数据起始点**：数据存放在 `keccak256(p)` 开始的区域。

- **布局逻辑**：从起始点开始，数据按照静态数组的规则排布（支持紧凑打包）。

- **多维数组（递归）**：如果是 `uint[][]`，那么每一个元素本身又是一个动态数组，递归应用哈希规则。

## 映射 (Mappings)

- **占位符 (Slot $p$)**：这个位置是**空**的，仅仅为了保证即使两个映射相邻，它们的哈希基数（$p$）也不同。

- **寻址公式**：`keccak256(h(k) . p)`。
  
  - $k$ 是键（Key），$p$ 是映射在合约中的槽位编号。
  
  - `.` 是拼接操作。
  
  - $h$ 是对键的填充函数（值类型补齐 32 字节，字符串/字节数组不补齐）。

### 结构体(Struct)

`struct` 并没有自己的哈希函数。它的位置完全取决于它被定义在哪里。

- **如果 Struct 是普通状态变量**：它的基准位置（Slot $p$）是按照[之前的顺序布局规则](https://docs.soliditylang.org/en/v0.8.20/internals/layout_in_storage.html)确定的。

- **如果 Struct 在 Mapping 里**：它的基准位置由 `keccak256(key . p)` 决定。

- **如果 Struct 在 Array 里**：它的基准位置由 `keccak256(p) + index * struct_size` 决定。

一旦基准位置确定了，结构体内部的成员就按照 **0, 1, 2...** 的偏移量往后排。

我们用一个代码例子来对比三种情况：

```solidity
contract StructDemo {
    struct S {
        uint256 a; // 偏移 0
        uint256 b; // 偏移 1
    }

    // 情况 1：直接作为状态变量
    S public s1; // 占据 Slot 0 和 Slot 1 (p=0)

    // 情况 2：在数组里
    S[] public sArray; // sArray 长度存在 Slot 2 (p=2)

    // 情况 3：在映射里
    mapping(uint => S) public sMap; // sMap 占位符在 Slot 3 (p=3)
}
```

#### 情况 1：静态寻址 (直接变量)

- `s1.a` 的地址就是 `0`。

- `s1.b` 的地址就是 `0 + 1 = 1`。

- **逻辑**：非常简单，就像两个独立的 `uint256` 变量一样。

#### 情况 2：数组寻址 (基准 + 偏移)

我们要找 `sArray[2].b`：

1. **找到数组起始点**：`start = keccak256(2)`。

2. **定位到索引 2 的结构体头部**：每个 `S` 占 2 个槽，所以 `sArray[2]` 的头部是 `start + 2 * 2 = start + 4`。

3. **加上成员 `b` 的内部偏移**：`b` 在结构体里是第二个成员（偏移为 1）。

4. **最终地址**：`keccak256(2) + 4 + 1`。

#### 情况 3：映射寻址 (哈希 + 偏移)

我们要找 `sMap[456].b`：

1. **找到结构体的基准点**：`base = keccak256(uint256(456) . uint256(3))`。

2. **加上成员 `b` 的内部偏移**：`b` 的偏移是 1。

3. **最终地址**：`base + 1`。

---

#### Struct 内部的“紧凑打包”

`struct` 内部的成员同样遵循紧凑打包规则。

```solidity
struct S2 {
    uint128 x; // 16 字节
    uint128 y; // 16 字节
    uint256 z; // 32 字节
}
```

在这个结构体中：

- `x` 和 `y` 会被挤在**同一个 Slot** 里（偏移量为 0）。

- `z` 会开启**下一个 Slot**（偏移量为 1）。

**寻址公式依然是：`基准地址 + 内部 Slot 偏移`。**

### 深度代码案例拆解：嵌套映射与结构体

这是文档给出的最强案例，我们来手算 `data[4][9].c` 的地址。

```solidity
contract C {
    struct S { 
        uint16 a; // 2 字节
        uint16 b; // 2 字节 (与 a 打包在 Slot 0)
        uint256 c; // 32 字节 (由于上一层没空了，开启 Slot 1)
    }
    uint x; // 占用 Slot 0
    mapping(uint => mapping(uint => S)) data; // 占用 Slot 1
}
```

#### 计算步骤：

1. **确定 $p$**：变量 `data` 之前只有 `x`，所以 `data` 的占位槽 $p = 1$。

2. **定位第一层键 `data[4]`**：
   
   - 它是映射里的映射。
   
   - 位置 = $Slot1\_child = keccak256(uint256(4) \cdot uint256(1))$。

3. **定位第二层键 `data[4][9]`**：
   
   - 这是结构体 `S` 的起始位置。
   
   - 位置 = $S\_start = keccak256(uint256(9) \cdot Slot1\_child)$。

4. **定位结构体成员 `.c`**：
   
   - 结构体内部：`a` 和 `b` 挤在第 0 槽，`c` 在第 1 槽。
   
   - 最终位置 = $S\_start + 1$。

## bytes与string：极度精妙的“短优化”

Solidity 对字符串做了非常聪明的处理，避免了短字符串也去消耗昂贵的哈希计算。

#### 情况 A：短字符串 (<= 31 字节)

- **存储位置**：直接存在占位槽 $p$ 中，不需要去哈希。

- **布局**：
  
  - **高位（左侧）**：存字符串内容。
  
  - **最低字节（最后 1 字节）**：存储 `length * 2`。

- **为什么是 `* 2`？**：因为最低位（LSB）必须是 `0`，用来标记这是“短模式”。

#### 情况 B：长字符串 (>= 32 字节)

- **存储位置**：槽 $p$ 存 `length * 2 + 1`（最低位为 `1` 标记为“长模式”）。

- **内容位置**：存在 `keccak256(p)` 开始的区域。

---

## 为什么映射不能迭代？

看完了寻址公式 `keccak256(k . p)`，你就明白了：

- 数据是根据 $k$ 随机分布在全链 $2^{256}$ 个插槽里的。

- 由于哈希是单向的，如果你不知道所有的 $k$，你就永远找不到它们存在哪。

- **结论**：映射在底层只是一个“查询表”，而不是一个“列表”。

---

## 给开发者的硬核提示

1. **存储冲突隐患**：虽然 $2^{256}$ 空间很大，但理论上哈希可能碰撞。在高度安全的合约中，要意识到这一点。

2. **空位浪费**：映射的占位槽 $p$ 永远是空的。如果你定义了 `mapping` 但从未使用，你依然在合约的静态存储布局里消耗了一个插槽位置。

3. **Panic(0x22)**：如果你通过 IR 编译，且手动修改了存储（比如通过汇编），结果导致字符串的长度标记位和实际存储不匹配，调用时会触发 `Panic(0x22)`。
