# 关于多个进程绑定端口

### 一、 内核层：多个进程到底能不能绑定同一个端口？

**答案是：在现代 Linux 内核中，完全可以。**

正如你看到的视频所说，关键在于一个 Socket 选项：`SO_REUSEPORT`（注意：是 `REUSEPORT` 端口重用，而不是传统的 `REUSEADDR` 地址重用）。

在 Linux 3.9 固件之后引入了 `SO_REUSEPORT` 特性。它的工作原理是这样的：

1. **真正的多进程绑定**：你可以启动 4 个独立的 Worker 进程，每个进程都独立创建 Socket，并调用 `bind(:8080)`。只要大家都开启了 `SO_REUSEPORT`，Linux 内核**不仅允许它们同时绑定成功，还会在内核层做一个负载均衡**。

2. **内核分发**：当一个 TCP 连接请求（SYN）到达 8080 端口时，Linux 内核会通过哈希算法（通常基于四元组：源IP、源端口、目的IP、目的端口），把这个连接**自动分发**给这 4 个进程中的某一个。

**传统做法（比如 Nginx）**：Nginx 的主进程绑定端口，然后通过 `fork` 把 Socket 句柄传给子进程，大家一起争抢连接（容易引发惊群效应）。而现在最新的 Nginx 也可以直接配置 `listen 8080 reuseport;`，让每个 Worker 进程独立去内核绑定端口。

### 二、 Node.js 层：为什么 Cluster 模式要“Hack” `listen`？

既然操作系统本身都支持 `SO_REUSEPORT` 了，为什么 Node.js 的 `cluster` 模块还要费尽心机地去伪造、去“Hack” `listen` 方法呢？

你的记忆力非常精准，Node.js 确实在底层做手脚。在 Node.js 中，当你用 `cluster.fork()` 启动多个 Worker 进程并让它们都执行 `server.listen(8080)` 时，**实际上只有 Master 进程真正绑定了 8080 端口，Worker 进程压根没有绑定。**

Node.js 之所以这么做，有以下两个核心原因：

#### 1. 历史原因与跨平台兼容性

Node.js 诞生和设计 Cluster 模式的时间非常早，当时的 Linux 还没普及 `SO_REUSEPORT`（或者还很不成熟），而且 Node.js 还要兼容 Windows（Windows 的网络机制和 Linux 大不相同）。为了实现一套代码跨平台跑多进程调度，Node.js 必须在应用层自己搞定负载均衡。

#### 2. 内核分发不均与“调度权”问题

这是最核心的工程考量。**Linux 内核的 `SO_REUSEPORT` 分发算法是盲目的（基于四元组哈希）**。内核不知道哪个 Worker 进程现在很忙（比如正在做密集的 CPU 计算），哪个 Worker 很闲。这会导致连接分发不均——某个 Worker 被活活累死（Event Loop 阻塞），而另一个 Worker 闲得发慌。

Node.js 为了追求极致的事件驱动效率，希望把**调度的绝对控制权**握在自己手里。

### 三、 Node.js Cluster 的两种工作模式

因为控制权在自己手里，Node.js Cluster 在内部玩出了两种花样：

#### 方式 A：Round-Robin 模式（轮询模式，Node.js 默认）

- **Master 进程**：独自霸占并绑定 `8080` 端口，专门负责当“前台接待”。

- **Worker 进程**：它们调用 `listen(8080)` 时，Node.js 底层拦截了这个调用，根本不向操作系统申请端口，而是向 Master 注册自己。

- **流程**：当一个新连接进来，Master 进程先接受（Accept）这个 Socket 句柄，然后通过内部的进程间通信（IPC），像发传真一样把这个 Socket 句柄按照轮询（A -> B -> C -> A）的顺序发给闲置的 Worker 去处理。

#### 方式 B：共享 Socket 模式（抢占模式）

- Master 进程创建好监听的 Socket，但不接受连接。它把这个 Socket 句柄直接发送给所有的 Worker。

- 所有的 Worker 进程共同监听这个由 Master 传过来的同一个 Socket。

- **流程**：新连接进来时，由操作系统内核来决定哪个 Worker 抢到这个连接（类似传统 Nginx 靠操作系统内核去盘活）。但这种模式在 Node.js 里会导致严重的负载不均，所以默认是不启用的。

# 内核用的哈希算法

在 Linux 内核通过 `SO_REUSEPORT` 分发 TCP 连接时，为了保证高效和公平，采用的是**四元组哈希算法**。

这里的“四元组”指的是能够唯一确定一个网络连接的四个核心元素：

1. **源 IP**（Source IP）：客户端的 IP 地址。

2. **源端口**（Source Port）：客户端的端口号。

3. **目的 IP**（Destination IP）：服务器的 IP 地址。

4. **目的端口**（Destination Port）：服务器的端口号。

### 一、 核心算法流程

内核的计算过程可以拆解为以下三个步骤：

#### 1. 提取并计算原始哈希值（以经典的 Jenkins Hash 或 MurmurHash 为例）

内核会把这四个字段的二进制数据组合成一个字节流，然后通过内核自带的高效哈希函数（通常是 `jhash_3words` 或类似的变种）计算出一个 **32 位的无符号整数（u32）**。

Hash = Jenkins\_Hash(源IP,  目的IP,  源端口 | 目的端口)

> 💡 **为什么要引入扰动（Hash Seed）？**
> 
> 为了防止黑客恶意制造哈希冲突（即所谓的 DOS 攻击，通过构造特定的 IP 和端口让流量全部涌向某一个固定的 Worker 进程），Linux 内核在计算哈希时，还会引入一个系统启动时随机生成的**随机数种子（Random Seed）**。这样黑客就无法在外部反推系统的哈希规律。

#### 2. 取模映射（定位到具体的进程）

算出了 32 位的哈希值后，内核需要把它映射到当前绑定在该端口的进程列表上。假设当前有 $N$ 个进程开启了 `SO_REUSEPORT` 绑定在同一个端口，内核会直接对进程数量进行取模（Modulo）运算：

Index = Hash mod N

算出来的 `Index`（比如 0、1、2... $N-1$），就代表这个连接将由第几个进程的监听队列（Accept Queue）来接收。
