# 一致性哈希

一致性哈希（Consistent Hashing）最核心的运作机制。

用“哈希环”来做比喻是最形象的。

### 完美验证你的理解：哈希环机制

正如你所说，传统取模（`Hash % N`）的致命缺点是“牵一发而动全身”。而一致性哈希通过引入**虚拟圆环**完美解耦了这一点：

1. **映射服务器**：把一个圆环分成 $2^{32}$ 个点（从 $0$ 到 $2^{32}-1$）。把服务器节点的名称（如 IP）进行哈希计算，算出的值就是它在环上的**具体位置**。

2. **映射数据（Cache Key）**：当你要存取缓存时，同样算出 `Hash(Key)`，把这个 Key 也扔到这个环上。

3. **顺时针查找（最近节点）**：从 Key 的位置出发，**沿着圆环顺时针往前走，碰到的第一台服务器**，就是它要存放或读取的节点。

#### 验证你的结论：

- **节点崩了/缩容**：假设 B 节点崩了，原先落在 B 上的缓存 Key 再顺时针找，就会找到 C 节点。**确实只有原本属于 B 的那一部分缓存会失效（需要去 C 重新加载）**，而 A、C、D 节点上的缓存雷打不动。这就把损失降到了最低。

### 现实很残酷：一致性哈希的致命缺陷与“补丁”

虽然你的理解在理论上无懈可击，但在实际工程中，如果只按上面的简版环来跑，会遇到一个极其严重的现象——**数据倾斜与雪崩（Data Skew & Avalanche）**。

#### 1. 痛点：数据不均匀（数据倾斜）

在环上，我们很难保证几台服务器的哈希值刚好“平分”这个圆环。

假设你有 3 台服务器，哈希出来的位置刚好都挤在环的左半边。结果就是：**圆环上 80% 的区域顺时针走都会撞到 A 服务器，A 被活活累死，B 和 C 闲得发慌。**

#### 2. 解决方案：引入“虚拟节点”（Virtual Nodes）

为了解决这个问题，现代所有的一致性哈希算法（比如 Memcached、Cassandra，或者我们前面提到 Linux 内核的 eBPF 调度）都引入了**虚拟节点**的概念。

- **做法**：不直接把物理服务器扔到环上。而是给每台服务器分出 100 个甚至 200 个“分身”。
  
  - 比如服务器 A，我们在环上建立 `A#1`、`A#2`、`A#3`...`A#100`。
  
  - 服务器 B，建立 `B#1`、`B#2`...`B#100`。

- **效果**：当成百上千个虚拟节点密密麻麻、均匀地散落在整个哈希环上时，任何一个缓存 Key 扔上去，抢到流量的概率就几乎完全均等了。

- **映射回物理机**：如果 Key 顺时针碰到了 `B#42`，内核或中间件知道这只是 B 的分身，就会把请求转交给真正的**物理机 B**。

<br />

一旦某台物理机挂了，它在哈希环上对应的所有虚拟节点必须在第一时间被同步下线（从环上移除）。

如果不清除，就会发生严重的灾难。我们来看看如果不清除会发生什么，以及清除之后流量是如何完美被分摊的：

### 1. 如果不清除：请求会全部“石沉大海”

假设物理机 B 挂了，但它在环上的 100 个虚拟节点（`B#1`, `B#2` ...）还在。
当客户端发来请求，通过哈希计算顺时针找到了 `B#42` 这个虚拟节点，路由系统一看：“哦，是 B 的分身，那把请求发给物理机 B 吧！”

然而物理机 B 已经死机了。结果就是：**所有命中 B 分身的流量，全都会连接失败报错，导致服务局部瘫痪。** 虚拟节点必须和物理机的生命周期死死绑定。

### 2. 清除之后：见证一致性哈希最精妙的“流量稀释”

把挂掉的物理机 B 的虚拟节点全部清除后，最神奇的事情发生了：原先打向 B 的流量，并不会一股脑全部压给某一台幸存的机器，而是被极其均匀地“稀释”分摊给了剩下的所有物理机。

我们来还原一下清除后的现场：

- 假设环上原本交错排列着：`A#1` -> `B#1` -> `C#1` -> `A#2` -> `B#2` -> `C#2`

- 现在 B 挂了，系统把 `B#1` 和 `B#2` 抹去。

- 环上的顺序变成了：`A#1` -> `C#1` -> `A#2` -> `C#2`

发现了嘛？

- 原本在 `A#1` 和 `B#1` 之间的请求，因为 `B#1` 消失了，顺时针继续走，会撞到 **`C#1`**（这部分流量给了 C）。

- 原本在 `C#1` 和 `B#2` 之间的请求，因为 `B#2` 消失了，顺时针继续走，会撞到 **`A#2`**（这部分流量给了 A）。
